Основной high-ROI этап завершён

Закрыты наиболее значимые проблемы с административным доступом, правильными ответами, результатами тестов, оплатой, редактируемым HTML и правами записи в базу.

Что было исправлено

Было Что сделано Результат
Общий пароль администратора хранился в базе, а вход отмечался в браузере. Вход переведён на Supabase Auth с e-mail, паролем и серверной проверкой роли администратора. Нельзя открыть админку простой подменой данных в браузере или прочитать общий пароль из базы.
Анонимные посетители имели слишком широкие права на таблицы. Включены строгие правила доступа — RLS. Посетитель не может изменять тесты, вопросы, настройки и чужие результаты.
Правильные ответы можно было получить вместе с вопросами. Публичный API отдаёт вопросы без correct_answer, а подсчёт выполняется на сервере. Нельзя заранее скачать ответы или подделать итоговый балл на стороне браузера.
Для доступа к результату было достаточно узнать ID сессии. Каждой сессии выдаётся отдельный случайный access_token. Зная только адрес или ID, нельзя прочитать чужой результат, e-mail и ответы.
Оплату можно было попытаться создать для чужой сессии. Checkout проверяет пару sessionId + accessToken. Нельзя связать оплату с чужим результатом без секретного токена этой сессии.
Отправку результатов по e-mail можно было вызвать напрямую. Функция принимает запросы только от доверенной серверной части. Посторонний не сможет использовать её для рассылки или получения чужих данных.
Редактируемый HTML мог содержать исполняемый код. HTML по умолчанию отображается как текст, а форматирование обрабатывается безопасным парсером. Закрыта XSS-уязвимость: скрипты и обработчики событий не исполняются.
Загрузка и удаление изображений были слишком открыты. Публичным оставлен только просмотр; изменение разрешено администратору. Посетитель не сможет заменить или удалить изображения вопросов.
В логах и аналитике могли появляться лишние идентификаторы. Убрано логирование e-mail и передача ID тестовой сессии как идентификатора транзакции. Снижен риск утечки персональных данных через служебные системы.
В проекте оставались старые функции и чувствительные служебные файлы. Устаревшие функции и временные артефакты удалены, примеры конфигурации очищены. Уменьшена общая поверхность атаки.
Защитные проверки выполнялись вручную. Security-регрессии, безопасное форматирование, линтер и сборка добавлены в публикацию. Опасные части сложнее случайно вернуть будущим изменением.

Что изменилось для пользователей

Прохождение теста

Аккаунт по-прежнему не требуется. Вопросы и отправка ответов работают по прежнему сценарию.

Оплата и результаты

Дополнительный защитный токен сохраняется браузером автоматически и не требует действий пользователя.

Ссылка на результат

После открытия секретная часть ссылки переносится во временное хранилище и удаляется из адресной строки.

Административная панель

Вместо общего пароля теперь используется настоящий аккаунт администратора с e-mail и паролем.

Безопасное форматирование контента

Основной режим

Произвольный HTML не исполняется. Поддерживаются безопасные абзацы, списки, разделители и выделение жирным.

Подготовленный откат

При проблемах со старым контентом можно временно включить режим совместимости. Он допускает небольшой набор HTML, продолжая удалять опасные теги и атрибуты.
